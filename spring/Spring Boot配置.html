# 自动配置

Spring Boot的自动配置是在应用程序启动时进行的。它决定Spring配置应该用哪个，不该用哪个。

向应用程序加入Spring Boot时，引入了一个名为spring-boot-autoconfigure的jar文件。此文件包含了许多配置类，每个配置类都在应用程序的Classpath里，都有机会为应用程序的配置添砖加瓦。

这些配置类利用了Spring4.0新增的条件化配置。条件化配置允许配置存在于应用程序中，但在满足某些特定条件之前都忽略这个配置。

# 自动配置的决策

结合之前创建的ReadingList项目，自动配置会做出以下配置决策：

  • 因为Classpath里有H2，所以会创建一个嵌入式的H2数据库Bean。

  • 因为Classpath里有Hibernate（Spring Data JPA传递引入），所以会自动配置与Hibernate相关的Bean。

  • 因为Classpath里有Spring Data JPA，所以会自动配置为根据仓库的接口创建仓库实现。

  • 因为Classpath有Thymeleaf，所以会将Thymeleaf配置为Spring MVC的视图。它对应的视图解析器会解析相对于Classpath根目录的/templates目录中的模板。

  • 因为Classpath里有Spring MVC，所以会配置DispatcherServlet并启用Spring MVC。

  • 因为这个一个Spring MVC Web应用程序，所以会注册一个资源处理器，把相对于Classpath根目录的/static目录中的静态资源提取出来。

  • 因为Classpath里有Tomcat，所以会启动一个嵌入式的Tomcat容器，默认监听8080端口。

# 自定义条件类

在Spring中实现Condition接口，重写其matches()便可编写自定义条件类。比如，编写JdbcTemplateCondition在Classpath里存在JdbcTemplate类时才会生效。

public class JdbcTemplateCondition implements Condition {
    @Override
    public boolean matches(ConditionContext context,
                    AnnotatedTypeMetadata metadata) {
        try {
            context.getClassLoader().loadClass(
                "org.springframework.jdbc.core.JdbcTemplate");
            return true;
        } catch (Exception e) {
            return false;
        }
    }
}

# 条件化注解

Spring Boot提供了如下条件化注解。

条件化注解 配置生效条件
@ConditionOnBean 配置了某个特定Bean
@ConditionOnMissingBean 没有配置特定的Bean
@ConditionOnClass Classpath里有指定的类
@ConditionOnMissingClass Classpath里缺少指定的类
@ConditionOnExpression 给定的Spring Expression Language表达式计算结果为true
@ConditionOnJava Java的版本匹配特定值或一个范围值
@ConditionOnJndi 参数中给定的JNDI位置必须存在一个，如果没有给参数，则要有JNDI InitialContext
@ConditionOnProperty 指定的配置属性要有一个明确的值
@ConditionOnResource Classpath有指定的资源
@ConditionOnWebApplication 这个一个Web应用程序
@ConditionOnNotWebApplication 这不是一个Web应用程序

# 自定义配置

影响自动配置的方式有：一，使用显式配置进行覆盖；二，使用属性进行精细化配置。

# 显式配置

大多数情况下，自动配置的Bean就能满足需要，但有些情况下，还是需要覆盖某些自动配置来更好地实现需求。比如，安全特性的自动配置。

想要覆盖Spring Boot的自动配置，首先编写一个显式的配置，然后降低自动配置的优先级。

Spring Boot Security默认的自动配置为SpringBootWebSecurityConfiguration。

@Configuration
@EnableConfigurationProperties
@ConditionalOnClass({ EnableWebSecurity.class, AuthenticationEntryPoint.class })
@ConditionalOnMissingBean(WebSecurityConfiguration.class)
@ConditionalOnWebApplication
@EnableWebSecurity
public class SpringBootWebSecurityConfiguration

# 属性配置

Spring Boot自动配置的Bean提供了数百个用于微调的属性。当你需要调整这些属性值时，只要在环境变量、Java系统属性、JNDI（Java Naming and Directory Interface）、命令行参数或属性文件中进行指定。

# 属性配置示例

将spring.main.show-banner属性设置为false来禁用ascii-art Banner。

可以在运行应用程序的命令行参数里指定参数--spring.main.show-banner=false。

可以创建一个名为application.properties的文件，包含内容spring.main.show-banner=false。

可以创建一个名为application.yml的YAML文件，内容如下：

spring:
	main:
		show-banner: false

# 属性源

Spring Boot能从多种属性源获得属性，包含如下几处：

  1. 命令行参数

  2. java:comp/env里的JNDI属性

  3. JVM系统属性

  4. 操作系统环境变量

  5. 随机生成的带random.*前缀的属性

  6. 应用程序以外的application.properties或application.yml文件

  7. 打包在应用程序内的application.properties或application.yml文件

  8. 通过@PropertySource标注的属性源

  9. 默认属性​

上面的列表是按照优先级排序。即/config子目录中的application.properties会覆盖应用程序Classpath里的application.properties中的相同属性。

同一优先级位置的application.yml的属性会覆盖application.properties的相同属性。

application.properties和application.yml文件能放在以下四个位置（按照优先级排序）：

  1. 外置，在相对于应用程序运行目录的/config子目录里

  2. 外置，在应用程序运行的目录里。

  3. 内置，在config包内。

  4. 内置，在Classpath根目录。

# Bean的外置属性

在属性文件（application.properties或application.yml）中定义自定义属性。比如custom.associateId=daking。

在代码中使用@ConfigurationProperties("custom")来引入以custom为前缀的外置属性。

注意，要向Spring配置类添加@EnableConfigurationProperties后，@ConfigurationProperties才会生效。而Spring Boot自动配置已经为我们添加了。

一般是创建一个单独的Bean来专门收集这些外置属性。

@Component
@ConfigurationProperties("custom")
class CustomProperties {

    lateinit var associateId: String
}

# Profile配置

Spring 3.1开始支持基于Profile的配置。它是一种条件化配置，基于运行时激活的Profile，会使用或忽略不同的Bean或配置类。

为特定的环境进行特定的Profile配置，可实现开发、测试、生产等不同环境的隔离。比如为生产环境制定的production Profile。

@Profile("production")
@Configuration
@EnableWebSecurity
public class SecurityConfig extends WebSecurityConfigurerAdapter {
    // ...
}

# application.properties与Profile

用application.properties来指定满足各种环境下的通用的属性配置。

遵循application-{profile}.properties命名格式来创建特定环境的属性文件。比如，创建一个仅用于生产环境的application-production.properties文件，在这个文件中指定属性配置。

# application.yml与Profile

application.yml的用法与上面的application.properties类似，但可以把所有的Profile配置都放在同一个application.yml文件中。

每段Profile配置用spring.profiles属性来指定Profile名称，若未指定则表示对全部Profile生效。不同的Profile配置之间用---作为分隔符。

注意，特定Profile配置会覆盖通用配置的相同属性。

logging:
    level:
        root: INFO

---

spring:
    profiles: development

logging:
    level:
        root: DEBUG

---

spring:
    profiles: production

logging:
    path: /tmp/
    file: BookWorm.log
    level:
        root: WARN
